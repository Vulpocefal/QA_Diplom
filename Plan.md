# План автоматизации тестов

## 1. Автоматизируемые сценарии

- Вызов форм с помощью кнопок «Купить» и «Купить в кредит»;
- Заполнение полей формы «Оплата по карте» валидными значениями с последующей отправкой по кнопке «Продолжить»;
- Заполнение полей формы «Оплата по карте» невалидными значениями с последующей отправкой по кнопке «Продолжить» в т.ч. проверка граничных значений;
- Заполнение части полей формы «Оплата по карте» невалидными значениями с последующей отправкой по кнопке «Продолжить»;
- Отправка пустой формы «Оплата по карте» по кнопке «Продолжить»;
- Заполнение полей формы «Кредит по данным карты» валидными значениями с последующей отправкой по кнопке «Продолжить»;
- Заполнение полей формы «Кредит по данным карты» невалидными значениями с последующей отправкой по кнопке «Продолжить» в т.ч. проверка граничных значений;
- Заполнение части полей формы «Кредит по данным карты» невалидными значениями с последующей отправкой по кнопке «Продолжить»;
- Отправка пустой формы «Кредит по данным карты» по кнопке «Продолжить».

## 2. Инструменты

- **Язык программирования**: Python. <br> Обоснование: Широкое распространение, большое количество библиотек для автоматизации тестирования, высокая производительность.

- **Фреймворк автоматизации**: Selenium. <br> Обоснование: Selenium — стандарт индустрии для веб-автоматизации, совместим с используемым в нашей автоматизации языком программирования, поддерживает разные браузеры.

- **Библиотека для написания тестов**: PyTest. <br> Обоснование: Представляет удобные аннотации, функции для группировки тестов, генерации отчетов.

- **Визуализация результатов тестирования**: Allure Report. <br> Обоснование: Представляет наглядные отчеты по результатам тестирования. Совместим с используемым в нашей автоматизации языком программирования, и легко интегрируется в GitHub Actions.

- **Система контроля версий**: Git. <br> Обоснование: Стандарт для совместной разработки, позволяет отслеживать историю изменений и версий.

## 3. Перечень и описание возможных рисков при автоматизации

- **Риск 1**: Изменение пользовательского интерфейса (UI). <br> Описание: Частые изменения в верстке или структуре приложения могут привести к «падению» автотестов (ложноотрицательные результаты), потребуют постоянного обновления селекторов и локаторов элементов. <br> Меры по минимизации: Использование устойчивых локаторов, модульная структура тестов.

- **Риск 2**: Нестабильность тестового окружения. <br> Описание: Медленная работа приложения, падения тестовой базы данных или сетевые задержки могут приводить к непредсказуемым результатам прогонов. <br> Меры по минимизации: Настройка мониторинга окружения, использование ожиданий (waits) в коде автотестов, повторные запуски (retry) упавших тестов.

- **Риск 3**: Нехватка времени/ресурсов. <br> Описание: Недооценка сложности задач может привести к срыву сроков написания автотестов. <br> Меры по минимизации: Детальное планирование и интервальная оценка трудозатрат с учетом рисков, приоритизация сценариев для автоматизации (сначала критическая бизнес-логика).
